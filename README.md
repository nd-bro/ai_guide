# 부동산 AI 챗봇 — 할루0 유지하면서 기능 살리기, 조언 요청 보고서

> **보내는 사람**: 김수현 (검단엘리트부동산 / 부동산 7년차, 어머니 25년차 — 각각 / 코드 비전문가)
> **받는 사람**: 개발자 오빠(문숙이) — **백엔드·관리서버 담당** (그래서 백엔드 기능 조언 받고 배우려고 함)
> **부탁**: 아래 우리 챗봇 **진행과정 + 보안점**을 보고, **할루시네이션 0%는 그대로 유지하면서 기능을 더 살리려면 어떤 모델·기법을 어떻게 써야 하는지** 상세히(보고서 식으로) 알려줘. 비용·유지보수 감당되는 선에서.

## 0. 한 줄 요약 (오빠가 5초만에)

우리 챗봇은 **할루(거짓말)는 거의 0%로 막는 데 성공**했어. 근데 너무 빡빡하게 막아서 **정당한 질문도 "확인이 필요해요"로 거부하는 일(과차단)이 많아** — 답을 더 잘 해주게(기능) 하면서도 **할루 0%는 깨지 않는** 방법이 궁금해. 오빠가 말한 "**강도 조절**"을 우리가 어떻게 구현하면 좋을지 구체적으로.

우선 결과론적으로 거의 0%를 막는건 불가능해. 이유는 LLM 모델은 기존의 여러 도메인을 학습하는 방식이고 이를 보안하기 위해 RAG(구조를 사용하기 때문)

<img width="1536" height="1024" alt="ab4086d2-ef18-4cb2-874f-2c9baddefc0e" src="https://github.com/user-attachments/assets/159e6393-aa35-43e0-bc6d-583736745816" />




우선 위의 그림에서 부동산 지식문서("법률 정보","규정"등)절대로 훼손되면 안되는 정보를 뽑아내서 찾아 낼때는 보통 LLM 질문에 답을 생성하기전 벡터(BOW, BM25, LLM2vec)등을 참조 해서 답변을 생성하고 이를 참조 하는데
기존에 openmodel에서는 이를 조절하기 어려움.

Sonnet을 기준으로 그나마 변경 할 수 있는 paramater가 있는데

Temperature(모델의 온도) : 0.0 ~ 0.2로 낮게 설정. temperature 낮을수록 모델이 창의적인 단어를 만들어내는 것을 막고, 검색된 문서에 있는 내용을 바탕으로 객관적이고 사실적인 답변을 생성.

(<img width="724" height="567" alt="image" src="https://github.com/user-attachments/assets/11a73394-cb11-438e-8045-4bbc9b43086c" />)



위와 같이 temperature를 조절할때 마다 모델이 출력하는 결과물의 위와 같음

Top-K : 10 ~ 30 정도로 설정하여 가장 연관성이 높은 상위 문서 청크(Chunk)만 모델에 전달. 너무 많은 문서를 넘기면 오히려 모델이 핵심 정보를 놓칠 수 있음.
(내가 생각할때 아래에 법률정보다 A급 판례 분석등이 있다고 하면 chunk 벡터는 많이 있는거 같은데 부동산 특성상 비슷한 구조의 단어조합과 배열이 많아서 몇개의 문서만 참조하는게 좋을듯)

Top-P:  0.0~0.3낮게 설정. 단어 선택의 확률 범위를 백분율(0~1)로 제한 이는 참조한 문서에서 중요한 단어만 생성하여 답변을 하게 설정.

하지만 이런 AI프로픔터(?) 개발 하는 사람들이 튜닝으로 잡는 방법인데 이런 모델의 구조를 어느정도 알고 왜 이렇게 작동하는지를 알아야 설명이 가능한데
비전공자 혹은 AImodel의 이해도가 얼마나 높은지 몰라서 설명하기가 힘듬.

sonnet을 제외하고도 타 다른모델도 해당 parameter가 있고 이걸 기반으로 보통 튜닝을 함






## 1. 우리 챗봇 진행과정 (오빠 llama3+doc 방식과 비교)

| 항목 | 오빠 챗봇 | 우리 챗봇 |
|---|---|---|
| 베이스 모델 | **llama3**(로컬) + doc 참조 + bow 기법 | **Claude(Anthropic API, sonnet)** — 자체 호스팅 안 함 |
| 지식 주입 | 문서(doc) 최대한 참조 | ① **검증된 지식카드(.md, 18섹션)** — ★**우리 법률 프로젝트(legal-ai)의 검증된 A급 판례 분석보고서를 참고·이식**해 생산(reviewer 검증완료·환각0) + 법제처 API로 글자 단위 재검증(판례번호·법조문 원문 1:1) ② **하이브리드 RAG = KURE 벡터(의미) + BM25(키워드=BoW, 오빠가 말한 "bow 기법") + 크로스인코더 리랭커를 RRF로 결합** ③ regulations.json(세금·대출 SSOT) |
| 할루 방어 | "강도 조절" + bow | **우리도 BM25(bow)는 이미 들어가 있음**(위 RAG) + **다층 가드 7종**(아래 §2) + 21개 공공기관 실측값만 + LLM이 못 만지는 "구조적 즉답"(카드·세금표) |
| ★bow(BM25) 비교 | doc + bow로 참조강도 조절 | **BM25(bow) 단독이 아니라 벡터+리랭커와 RRF 결합**(`hybrid_retrieval.py`). 즉 오빠 bow보다 한 단계 더 — 키워드(bow)+의미(벡터)+재정렬(리랭커) 3중 |
| 할루 현황 | 심해서 강도조절 중 | **거의 0%**(숫자·법조문·판례·단지명·주소는 공공기관 SSOT 1:1, 못 만들면 "[출처 확인 필요]"로 격리) |

**핵심 차이**: 오빠는 "doc 참조 강도"로 할루를 줄이고, 우리는 "**LLM이 숫자·법조문을 못 만들게 구조적으로 차단**(검증된 카드·표만 verbatim 서빙)"으로 할루를 0에 가깝게 만들었어. 그래서 **할루는 잡혔는데, 대신 너무 많이 거부(과차단)**해.

## 1-B. 백엔드 규모·기술 스택 (실측 2026-06-16 — 오빠가 백엔드 담당이라 상세히)

**코드 규모(실측, node_modules 제외)**:
- **elite 본체**: 백엔드 Python **14.2만줄** + 프론트(website) TS·TSX **20.1만줄** = **약 34만줄**
- **법률 AI 프로젝트**(별도 repo): TS **9.3만줄**
- **CRM·관리자 생태계**(코덱스 작업, admin-backend·admin-realtime·crm-desktop·superadmin): admin-backend Py 18만/TS 21만 등 — ★단 이건 **elite 베이스를 포함한 확장본**이라 단순 합산하면 중복(고유 추가분 위주로 봐야 함)

**백엔드 구조(실측)**:
- 라우터 **97개** / **API 엔드포인트 639개**(@router.get/post/...) / 백엔드 서비스 모듈 **123개** / **DB 테이블 59개** / Alembic 마이그레이션 7 / pytest 테스트 **227개** / 프론트 페이지 91개

**AI/할루0 기술 스택(★오빠 조언 핵심 — 누락 없이)**:
- 모델: **Claude(sonnet-4-6) API**. thinking(확장사고)·스트리밍 **미사용**(개선 여지).
- **동적 max_tokens**(음성300/인사400/일반900/법률1500) + **프롬프트 캐싱(ephemeral)** 으로 입력비 절감

- **구조적 즉답 fastpath 10종**(LLM 안 거쳐서 할루 원천 0): 검단단지사실·취득세·재산세(보유세)·관리비·인지세·중개보수·시세·개념카드·지식카드·조정대상여부
- **다층 가드 8종 모듈**: output_sanitizer·source_verifier·confidence_scorer(9신호+4VETO)·numeric_claim_validator·freshness_guard·agency_resolver·legal_date_guard·ai_safety_guard

- **서버 방어**: Redis 레이트리밋(전워커 공유)·버스트 표적재시도·다중에이전트 적대검증·SafeHandler

**지식 DB 규모(실측)**: 옵시디언 vault **26,620 노트(604MB)** + RAG 코퍼스 **174MB** + 공공데이터 70 JSON(6MB)

## 2. 할루0를 만드는 우리 가드 구조 (이게 강할수록 기능이 죽음)

답변 나가기 직전에 **7개 가드**가 한 번에 검사해서, 하나라도 걸리면 "확인이 필요해요" 안전문구로 바꿔버려:

1. **출력 위생(sanitizer)** — 금지표현·위험표현 차단
2. **출처 정렬(source alignment)** — 답변이 근거 문서와 안 맞으면 차단
3. **숫자 검증(numeric validator)** — 답변의 숫자가 주입한 공공기관 값에 없으면 "[확인필요]"로 바꿈 ← **★과차단 주범**
4. **신뢰도 점수(confidence) + VETO** — 인용 없으면/지역 안 맞으면 55점 상한 → 차단
5. **신선도(freshness)** — 공공기관 데이터가 오래되면 차단
6. **공공기관 출처 강제** — 13개 기관 출처 0건이면 차단
7. **법률 프록시 경로** — 법률·세금 질문은 별도 법률 AI로 보냄

**문제(우리가 실측한 것)**:
- 정당한 질문인데 답변에 "**제25조**" 같은 법조항 번호가 들어가면, 숫자검증(3번)이 "승인 안 된 숫자"로 보고 차단 → **법률 설명을 못 함**
- "보증금 안 돌려줘 어떻게?" 같은 **제일 중요한 고객 질문**도 답변 속 비율·숫자 때문에 거부
- 골든 테스트(48문항)에서 **응답률이 28% → 66.7%**까지 올라왔는데, 나머지는 이 과차단 때문에 막혀

→ 즉 **"강도"가 너무 세서 기능이 죽는 상태**. 오빠가 말한 강도조절이 정확히 이 문제야.

## 3. 오빠한테 묻고 싶은 것 (상세 조언 요청 — 이게 핵심)

### Q1. "강도 조절"을 우리는 어떻게 구현해야 해?
오빠는 doc 참조 강도를 조절한다고 했는데, 우리는 가드가 7개라 어디를 어떻게 풀어야 할지 모르겠어.
- **질문 종류별로 가드 강도를 다르게** 하는 게 맞아? (예: 시세·세율 숫자 = 빡빡하게 / 법률 설명·절차 안내 = 느슨하게)
- **법조항 번호(제25조)는 숫자검증에서 빼는** 화이트리스트 방식이 정석이야? 아니면 더 좋은 방법?
- 강도를 0~10으로 조절하는 **하나의 노브**로 만들 수 있어, 아니면 가드마다 따로?

### Q2. 할루0 유지하면서 기능 살리는 **모델·기법**은?
비용·유지보수 감당되는 선에서(회사들 수억짜리 모델은 우리가 못 씀):
- **Claude API 그대로 + 기법 보강**이 나아, 아니면 **작은 모델 파인튜닝(자체 호스팅)**이 나아? **★우리 도메인 자산 — 어머니 25년 + 나 7년(각각, 합산 아님) 검단 부동산 실무 데이터·노하우**(매물·상담·계약·분쟁 사례). 이 정도면 도메인 특화 파인튜닝 자산으로 충분할까? 충분하면 어떤 형태(카드 .md / Q&A쌍 / 상담 로그)로 가공해야 해?
- "**그라운디드 생성(grounded generation)**" / "**제약 디코딩(constrained decoding)**" 같은 기법이 우리 과차단을 줄여줄 수 있어?
- RAG를 **리랭커 임계값·검색 개수**로 강도조절 하는 게 우리한테도 맞아?
- 답변을 **2단계**(① 구조적 즉답=할루0 / ② 부족하면 LLM이 보강하되 출처검증)로 나누는 설계 어때?

### Q3. **파인튜닝 vs 프롬프트 vs 가드레일** — 우리 규모엔 뭐가 맞아?
- 우리는 1인 사업체라 **유지보수 적게** 가야 해. 셋 중 뭐가 비용·관리 대비 효율 좋아?

### Q4. 응답률 올리면서 할루 안 늘리는 **검증 방법**?
