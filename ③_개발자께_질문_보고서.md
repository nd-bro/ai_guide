# 부동산 AI 챗봇 — 개발자님께 추가로 여쭙는 보고서

> **보내는 사람**: 김수현 (검단엘리트부동산 / 코드 비전문가)
> **받는 사람**: 개발자님 (백엔드·관리서버 담당)
> **부탁**: 지난번 답변 정말 감사했습니다. 조언(temperature·BM25·Adaptive RAG·불확실성 명시)을 저희가 거의 다 채택하려고 합니다. 다만 답변 주실 때 **저희 실제 구조를 정확히 몰라 빗나간 부분**이 한 군데 있어서, 그 부분 구조를 알려드리고 **다시 구체적으로** 여쭙니다. 답해주시면 그대로 반영하겠습니다.

---

## A. 먼저 — 지난번 "구조를 봐야 안다"고 하신 부분, 구조를 알려드립니다

개발자님이 **"가드를 어떻게 쓰는지 코드를 정확히 봐야 이해할 수 있다"**, **"구조적 즉답 = 할루0 기법은 이해가 안 된다"**고 하셨습니다. 이 두 가지를 먼저 설명드립니다. (이게 정확해야 나머지 질문이 의미가 있어서요.)

### A-1. 저희 답변은 "2갈래(2-track)"로 나뉩니다

```
고객 질문
  ├─ Track A : 구조적 즉답  → LLM 안 거침. 검증된 값/카드를 "그대로" 꺼내 보여줌
  │     (세금·시세·중개보수·검단단지·지식카드 13장 + 인사·개념)
  │     → 숫자·법조항을 LLM이 "생성"하지 않으므로 그 필드 할루 = 원천 0
  │
  └─ Track B : RAG + LLM 답변 → 카드에 없는 질문만. 검색으로 근거 찾고 그 위에서 생성
        → 검색(KURE벡터 + BM25 + 리랭커 → RRF) → Claude 생성 → 가드 검사
```

- **"구조적 즉답"의 정체**: "AI가 0% 틀리게 만든다"가 **아니라**, 틀리면 안 되는 핵심필드(숫자·법조항)는 **AI에게 안 시키고**, 미리 법제처·공공기관에서 검증해 둔 값을 **그대로(verbatim) 꺼내** 보여주는 것입니다. 그래서 그 부분만큼은 할루가 태어날 자리가 없습니다.
- 개발자님 말씀하신 **"할루 0%는 불가능"**은 **Track B(자유 생성)에 대해서는 100% 맞습니다.** 저희도 거기선 0%를 주장하지 않습니다. 다만 Track A 덕에 "절대 틀리면 안 되는 숫자"는 구조적으로 막아둔 것입니다.

→ **질문 A**: 이 2-track 설계(틀리면 안 되는 건 LLM 안 시키고 그대로 꺼내기 / 설명은 LLM이 근거 위에서) 방향이 맞다고 보시나요? 더 나은 구조가 있을까요?

### A-2. 과차단(답이 죽는 현상)의 실제 위치 — 검색이 아니라 "생성 후 숫자검증"

개발자님이 **"후보정 조건이 너무 많아 답이 죽는다"**고 하신 진단은 **정확합니다.** 그게 저희 최대 문제(골든 테스트 응답률 66.7%)입니다.

다만 위치를 **"RRF(검색) 이후 LLM 필터링"**으로 보셨는데, 저희 코드를 실제로 열어보니 과차단은 거기가 아니라 **"LLM이 답을 생성한 *후* 의 숫자 검증 단계(`numeric_claim_validator`)"**에서 일어납니다.

- 동작: 답변 안의 숫자를, 시스템이 주입한 공공기관 값 토큰과 **글자 단위로 정확히(exact match)** 대조 → **그 토큰에 없으면 답 전체를 차단**합니다.
- 문제: 법제처 근거가 RAG에 있는 정당한 답이어도, 답 문장 속에 "제25조"·비율·금액이 들어가면 → **주입 토큰에 정확히 안 맞아서 → 답 전체가 "확인이 필요해요"로 교체**됩니다.
- 그래서 **"보증금 안 돌려주면 어떡하나요?"(가장 중요한 고객 질문)·특약·하자책임**이 지금 막혀 있습니다.

→ 그래서 개발자님 조언 중 **"vector+BM25만 써봐"(검색층 개선)**는 좋은 위생이지만, **주 병목인 이 검문소(생성 후 숫자검증)를 직접 고쳐야** 풀린다고 이해했습니다. 맞는지 봐주세요.

## B. 저희가 생각한 해법 — 이 방향이 맞을까요? (핵심 질문)

저희 법률 AI 프로젝트(별도 repo)에 이미 비슷한 문제를 푼 코드가 있어서, 그 원리를 부동산 챗봇에 가져오려 합니다. 검토 부탁드립니다.

### Q1. 숫자검증을 "부재 시 차단" → "모순 시에만 차단"으로 바꾸는 것
- **지금**: 답에 있는 숫자가 주입 토큰에 **없으면 차단**.
- **바꾸려는 것**: 숫자가 **"이건 법정 OO다"라고 못박았는데, 알려진 모든 값과 모순될 때만 차단**하고, 그 외(파싱 실패·일부 일치·단순 설명 숫자)는 **통과(fail-open)**. 그리고 **법조항 번호(제25조)는 애초에 "검증 대상 숫자"에서 제외.**
- → **이 방향이 할루0를 깨지 않으면서 과차단을 푸는 정석이 맞나요?** 더 안전한 대안이 있을까요? (저희가 제일 두려운 건 "과차단 풀다가 진짜 가짜 숫자가 새는 것"입니다.)

### Q2. 차단 방식 — "답 전체 교체" 대신 "위험한 문장만 [확인 필요] 표시"
- 지금은 한 군데만 걸려도 답 전체를 안전문구로 바꿉니다. 개발자님 예시 프롬프트("한계를 명확히 밝힌다")처럼, **위험 문장만 [확인 필요] 표시하고 나머지는 답하는** 방식으로 바꾸려 합니다.
- → 이게 할루 관점에서 위험하진 않을까요? (부분 답변이 오히려 오해를 줄 위험 vs 거부보다 나음 — 어느 쪽이 맞을까요?)

### Q3. "모호 → 정밀 뻥튀기" 탐지
- "약 3억 정도"라고 물었는데 AI가 "3억 2,847만원"으로 단정하는 걸 잡고 싶습니다. (입력에 없던 정밀 숫자가 근거 없이 새로 등장하면 탐지)
- → 이런 "정밀도 인플레이션" 탐지가 실효성 있나요? 흔한 기법인가요?

## C. 모델·파라미터 (개발자님 전문 영역)

### Q4. Temperature — 저희는 지금 **설정 안 하고 있습니다(기본값)**
- 코드를 열어보니 Claude 호출에 `temperature`가 아예 안 들어가 있었습니다. 개발자님 조언대로 `0.1` 정도로 낮추려 합니다.
- → Sonnet 기준 **권장 값**이 0.1이 맞나요? `top_p`도 같이 낮추는 게 좋을까요(Sonnet은 top_k를 안 받아서요)? **이게 과차단과는 별개(위생)**라고 이해했는데 맞나요?

### Q5. Adaptive / Corrective RAG — 1인 사업체가 유지보수 감당 가능할까요?
- 근거 부족 시 차단 대신 재검색·질문 재작성 하는 방향, 좋아 보입니다. 다만 개발자님도 **"느리다"**고 하셨고, 저흰 1인이라 유지보수가 부담입니다.
- → 이걸 **Claude API 위에 얹는 정도**로 가능한가요, 아니면 **별도 오케스트레이터(LangGraph 등)**가 필요한가요? 1인 유지보수 기준 **현실적인 최소 구현**은 어디까지일까요?

### Q6. 파인튜닝 vs 프롬프트 vs 가드레일 — 데이터 "형태"가 중요하다 하셨는데
- 개발자님: *"data와 연수는 중요하지 않고, 어떻게 학습할 수 있는지 **데이터의 형태**가 더 중요한데 글만으로는 알 수 없다"* — 이 부분을 더 듣고 싶습니다.
- 저희 자산: ① 검증된 지식카드 13장(.md, 법제처 원문 1:1) ② 어머니 25년 + 저 7년 검단 실무(상담·계약·분쟁 사례) ③ 고객 Q&A 로그.
- → 만약 파인튜닝을 한다면 **어떤 형태**(카드 .md? Q&A 쌍? 지시-응답 형식?)로 가공해야 하나요? 아니면 **저희 규모(1인)엔 파인튜닝보다 RAG+가드 보강이 낫다**고 보시나요?

## D. 평가·보안

### Q7. 응답률 올리면서 할루 안 늘었는지 — 자동 평가 방법
- 지금은 골든 48문항을 사람이 눈으로 봅니다. **함정 질문(가짜 단지·투자단정)에 0% 틀리기**를 유지하면서 정당한 질문 응답률만 올리는 걸 검증하고 싶습니다.
- → LLM-as-judge(다른 모델이 채점) / 정답셋 회귀 / 둘 다? 1인이 굴릴 수 있는 **현실적 평가 셋업** 추천 부탁드립니다.

### Q8. 보안 — 외부 모델에 우리 DB가 새지 않게
- 저희 최대 자산: 고객 PII + 매물/고객 CRM DB + 판례·실무철학 DB.
- → 외부 API(Claude·파인튜닝)에 학습/전송 시 **유출 위험을 어떻게 막나요**? 자체 호스팅(로컬 모델)이 보안상 확실히 나은가요, 아니면 API의 "학습 미사용(zero-retention)" 설정으로 충분한가요? 백엔드 담당이시니 이 부분 꼭 듣고 싶습니다.

## E. 요약 — 개발자님께 핵심만

1. **진단 정확하셨습니다**: 후보정(가드)이 많아 답이 죽는 게 맞습니다. 위치만 검색→**생성 후 숫자검증**으로 정정해 이해했습니다.
2. **조언 거의 다 채택**: temperature(우리 미설정이었음)·BM25 하이브리드(이미 있고 OFF였음, 켜기만)·Adaptive RAG·불확실성 명시 프롬프트.
3. **핵심 질문**: 숫자검증을 "부재 차단→모순 차단"으로 바꾸는 게(Q1) 할루0를 안 깨는 정석인지, 그리고 1인 유지보수 기준 Adaptive RAG 최소 구현(Q5)·파인튜닝 데이터 형태(Q6)·보안(Q8)을 여쭙니다.
